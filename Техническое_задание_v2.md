# 1. Цель системы

Автоматизировать процесс проведения Performance Review сотрудников, включая:

- сбор оценок от разных участников
- использование матрицы компетенций
- расчёт итогов
- формирование отчётов

# 2. Общая логика процесса

Процесс запускается HR и проходит через последовательные этапы с возможностью контроля статусов.

**Основные этапы:**

- Актуализация списков участников — HR
- Актуализация списков команд — HR
- Актуализация матрицы компетенций — Лиды направлений/Руководитель (импорт из Excel)
- Оценка эффективности сотрудника (Performance) — Продакт-оунер
- Самооценка — Сотрудник
- Оценка hard skills — Лиды направлений/Руководитель
- Оценка СРВ (Сложность решения вопросов) — Лид направления/Руководитель
- Подведение итогов (1-on-1) — Руководитель + HR

> **Soft Skills 360° отложен на следующий этап развития системы.**

# 3. Роли в системе

- **HR (администратор процесса)** — полный доступ, управление всеми справочниками и циклами
- **Продакт-оунер** — оценка Performance сотрудников своих команд
- **Лид направления / Руководитель** — оценка Hard Skills и СРВ, проведение 1-on-1
- **Сотрудник (оцениваемый)** — заполнение самооценки

👉 Важно предусмотреть:

- один человек может иметь несколько ролей (например, Руководитель + Сотрудник)
- роли задаются как системные (постоянные) и не привязаны к конкретному циклу
- основная роль + до 3 дополнительных ролей у одного пользователя

# 4. Сущности (что должно храниться в системе)

## 4.1. Сотрудники

- ФИО
- email
- должность (job title — Frontend developer, QA engineer и т.д.)
- системные роли (массив: employee / manager / po / hr)
- команда / проект
- направление (Backend, Frontend, QA, Data, Mobile)
- руководитель (ссылка на другого сотрудника)
- продакт-оунер (ссылка на другого сотрудника)
- грейд (Jun1...Sen3 или TechLead1-3)
- **статус**: active / inactive / archived

## 4.2. Цикл оценки (Performance Review Cycle)

- название (например: Q2 2026)
- дата начала / окончания
- статус:
  - draft (черновик, ещё не запущен)
  - active (идёт оценка)
  - completed (завершён)
  - cancelled (отменён)
- список участников (с возможностью исключить отдельных сотрудников через чекбоксы)

## 4.3. Матрица компетенций

- компетенция
- тип:
  - hard
  - СРВ
- направление (для Hard Skills) — общая для СРВ
- описание компетенции
- описания шкалы 0/1/2/3 (текстовые)
- **подуровни (настраиваемые)** — обычно 9 (Jun1-Sen3) или 12 (с TechLead1-3)
- требование к каждому подуровню (балл 0-3)
- источник: импорт из Excel или ручное создание

**Импорт матрицы из Excel:**
- 2 листа в файле:
  - «Матрица [Направление]» — таблица компетенция × подуровни с баллами
  - «Описание [Направление]» — описание компетенции + описания шкалы 0/1/2/3
- Импорт **заменяет** матрицу выбранного направления полностью
- Количество подуровней берётся из файла автоматически

## 4.4. Оценки

- сотрудник
- кто оценивает
- цикл оценки
- тип оценки:
  - performance
  - hard skills
  - срв
- компетенция (для Hard / СРВ)
- оценка (число 0-3 для Hard/СРВ, Допуск/Недопуск для Performance)
- комментарий
- статус: draft / submitted / approved

## 4.5. Запрос на согласование (эскалация Недопуска)

- кто инициировал (ПО)
- сотрудник
- цикл
- обоснование Недопуска
- решение Руководителя: поддержал / не согласен + комментарий
- финальное решение HR: утвердить Недопуск / вернуть к допуску + комментарий
- статус маршрута: pending / approved / rejected

# 5. Детальная логика этапов

## Этап 1. Актуализация участников (HR)

**Функционал:**

- загрузка списка сотрудников из Excel
- ручное добавление сотрудника
- редактирование данных сотрудника
- управление статусами (Активный / Неактивный / Архивный)
- удаление дубликатов

**Логика статусов:**

🟢 **Активный** — участвует в новых циклах, виден во всех списках  
🟡 **Неактивный** — временно не участвует (декрет, длительная командировка), оценки сохраняются  
🔴 **Архивный** — финальный статус (уволился), вернуть в Активный нельзя

**Переходы:**
- Активный ⇄ Неактивный (двусторонний переход вручную HR)
- Активный/Неактивный → Архивный (финальный, необратимый)
- ❌ Архивный → Активный — невозможно

**При массовой загрузке Excel:**
- Если сотрудника НЕТ в новом файле — HR выбирает действие:
  - 🟡 Перевести в «Неактивные» (рекомендуется, по умолчанию)
  - 🔴 Архивировать
  - 🟢 Оставить «Активными»

**Оценки в завершённых циклах** сохраняются как есть в истории независимо от смены статуса.

## Этап 2. Актуализация матрицы (HR/Лид)

**Функционал:**

- импорт матрицы из Excel (2 листа: Матрица + Описание)
- ручное создание/редактирование компетенций
- импорт компетенции из другого направления (для переиспользования)
- выбор фильтра по направлению (Backend/Frontend/QA/Data/Mobile)

**В карточке компетенции:**
- название и описание
- описания шкалы 0-3 (для оценщика)
- требуемый балл по подуровням (для Hard Skills и СРВ)

**Внизу страницы — сводная таблица по направлению:**
- список всех компетенций × подуровни
- баллы окрашены цветом (0-красный, 1-жёлтый, 2-бирюзовый, 3-зелёный)
- последняя строка «Итого» — сумма баллов по всем компетенциям для каждого подуровня

👉 Важно:
- версия матрицы фиксируется на момент запуска цикла
- изменения после старта не влияют на текущий цикл

## Этап 3. Оценка эффективности Performance (Продакт-оунер)

### Форма оценки

5 критериев работы:
1. Закрытие задач в срок
2. Качество выполнения
3. Инициативность
4. Командное взаимодействие
5. Соблюдение процессов

**По каждому критерию:**
- описание критерия
- поле комментария оценщика (без баллов и без Допуск/Недопуск на уровне критерия)

**Итоговое решение** (в конце формы):
- **Допуск** к дальнейшей оценке → форма сохраняется напрямую
- **Недопуск** → запускается **эскалация согласования**

### Эскалация Недопуска

При выборе «Недопуск» ПО:
1. Появляется блок «⚠️ Решение требует согласования»
2. Обязательное поле «Обоснование недопуска»
3. Визуальный маршрут: **ПО → Руководитель → HR**
4. При отправке: статус сотрудника → «На рассмотрении»

**У Руководителя в дашборде** появляется блок «Запрос на согласование от ПО»:
- кнопка «Рассмотреть» → модальное окно
- Руководитель видит **полные оценки ПО по 5 критериям + обоснование**
- Решение: «Поддерживаю Недопуск» / «Не согласен (вернуть к допуску)»
- Обязательный комментарий
- Отправка в HR

**У HR на главной странице** появляется блок «Финальное согласование Недопуска»:
- HR видит **полную цепочку**: оценки ПО + решение Руководителя + комментарии
- Финальное решение: «Утвердить Недопуск» / «Вернуть к допуску»
- Обязательный финальный комментарий
- Уведомления отправляются всем участникам

## Этап 4. Самооценка (Сотрудник)

Форма состоит из 3 шагов:

### Шаг 1. Достижения за период

- Внедрение значимых проектов / крупных задач
- Нововведения и идеи с вау-эффектом
- Список достижений — до 6 пунктов (текстовые поля)

### Шаг 2. Выполнение плана развития

Таблица до 3 строк (план из прошлого цикла):
| № | Навык, который развивал | Комментарий |

### Шаг 3. Самооценка уровня

- текущий уровень (выбор из сетки)
- позиция: повысить / оставить / понизить
- обоснование (обязательное текстовое поле)

### Важно учесть

- форма доступна сотруднику только в рамках активного цикла
- сотрудник может сохранять черновик
- после отправки форма недоступна для редактирования
- HR и Лид видят заполненную самооценку только после её отправки

## Этап 5. Оценка Hard Skills (Лиды/Руководитель)

### Форма оценки

- список компетенций направления сотрудника (из матрицы)
- шкала 0-3
- описание каждой оценки по каждой компетенции (видно оценщику)
- комментарий по компетенции (опционально)

**Внизу формы — сводная матрица:**
- полная таблица: компетенции × подуровни (Jun1...Sen3 или больше)
- баллы окрашены цветом
- колонка «Факт» с фактической оценкой руководителя
- строка «Итого»: сумма баллов + сравнение с требованием уровня
- **3 индикатора сверху таблицы**: Факт / Требование уровня / Разрыв

**Логика индикаторов разрыва:**
- Зелёный: факт ≥ требование
- Жёлтый: разрыв 1-2 балла
- Красный: разрыв ≥ 3 баллов

👉 Важно:
- подтягиваются только компетенции направления сотрудника
- веса оценок не применяются — простой подсчёт суммы

## Этап 6. Оценка СРВ (Лид/Руководитель)

Аналогично Hard Skills, но компетенции **общие для всех направлений**:
- Самостоятельность
- Ответственность
- Коммуникация
- Командная работа
- Развитие и обучение
- Лидерство и менторинг

Также со сводной таблицей внизу и индикаторами Факт/Требование/Разрыв.

## Этап 7. Подведение итогов (1-on-1)

### Общая логика

После завершения всех этапов оценки Лид направления / Руководитель проводит встречу 1-on-1 с сотрудником **вне системы** (Outlook, личная договорённость). В ходе встречи или после неё Руководитель заполняет итоговую форму в системе. Финальное решение по грейду принимается совместно Руководителем и HR.

> 💡 Планирование встреч через Microsoft Teams / Outlook **не интегрировано** в систему. Сотрудник в своём профиле не видит блока встречи — он только видит свои задачи по оценке и в финале получит доступ к своей итоговой форме после её подтверждения Руководителем и HR.

### Итоговая форма — структура

#### Блок 1. Результаты оценки (заполняется автоматически системой)

**Простая таблица без весов:**

| Блок оценки | Оценщик | Этап | Итоговый балл |
|-------------|---------|------|---------------|
| Performance | Продакт-оунер | ✓ Допуск / ✕ Недопуск | Допущен / Не допущен |
| Hard Skills | Руководитель | Mid2 (грейд) | 14 / 24 (требование Mid2: 16) |
| СРВ | Руководитель | Mid2 (грейд) | 7 / 9 (требование Mid2: 6) |

**Под таблицей — 3 цветные карточки** для визуальной сводки.

👉 Важно: веса оценок и формула «итогового балла» убраны. Руководитель видит **простые итоги по каждому блоку отдельно** без расчётов.

#### Блок 2. Самооценка сотрудника (только для просмотра)

- Список достижений
- Самооценка уровня (текущий / позиция / обоснование)

#### Блок 3. Gap-анализ (заполняется автоматически)

| Компетенция | Тип | Текущий уровень | Ожидаемый уровень | Факт. оценка | Разрыв |

#### Блок 4. Оценка плана развития прошлого периода (Лид/Руководитель)

| № | Навык из прошлого цикла | Срок | Комментарий сотрудника | Статус: Выполнен/Частично/Не выполнен | Комментарий Лида |

#### Блок 5. Комментарии руководителя

- Общий комментарий по результатам
- Сильные стороны
- Зоны роста
- Рекомендации по развитию

#### Блок 6. Решение по грейду (Руководитель + HR)

**Базовая часть:**
- Текущий грейд (автоматически)
- Решение: **Повысить** / **Оставить**
- Комментарий

**Если «Повысить»:** Целевой грейд (выше текущего)  
**Если «Оставить»:** Решение по ЗП (Повысить ЗП / Оставить ЗП)

**Подтверждение:** Руководитель ✅ + HR ✅

#### Блок 7. План развития на следующий период (Руководитель)

| № | Навык / Направление развития | Срок | Комментарий |

# 8. Административная часть

## 8.1. Управление пользователями системы

### Аутентификация — Email + пароль

**Способ входа в систему:**
- Логин = **email** сотрудника
- Пароль — изначально временный (генерируется при создании учётной записи), потом сотрудник меняет на свой при первом входе

**Почему выбран Email + пароль:**
- ✅ Быстро разработать (1-2 дня)
- ✅ Не требует внешних интеграций (SSO, AD, OAuth)
- ✅ Работает с on-premise
- ✅ Полностью под контролем HR

> 💡 В будущей версии возможна интеграция с корпоративным SSO / Active Directory.

### Создание учётных записей — два способа

#### Способ 1: Импорт сотрудников из Excel (массово)

При импорте Excel-файла на странице «Сотрудники» HR может включить чекбокс **«Сгенерировать логины и пароли для новых сотрудников»** (включён по умолчанию).

После подтверждения импорта открывается окно **«🔑 Сгенерированные данные для входа»**:
- Таблица: ФИО | Email (логин) | Временный пароль | Копировать 📋
- Жёлтое предупреждение: «Сохраните этот список! После закрытия окна пароли уже не будут отображаться»
- 3 кнопки:
  - **↓ Скачать в CSV** (для рассылки или передачи)
  - **📧 Разослать каждому на email** (каждый сотрудник получает только свои данные)
  - **Готово** (закрыть окно)

#### Способ 2: Ручное добавление одного сотрудника

В форме «Добавить сотрудника» HR заполняет данные и видит чекбокс **«Сгенерировать логин и пароль»** (включён по умолчанию).

После нажатия «Добавить»:
- Если чекбокс включён → открывается окно **«🔑 Учётная запись создана»** с email и временным паролем (кнопки: 📧 Отправить на email / Готово)
- Если чекбокс снят → сотрудник добавляется в справочник без учётной записи (можно создать позже через «Пользователи»)

### Сброс пароля

В разделе **«Пользователи»** в каждой строке таблицы есть кнопка **🔑 «Сбросить пароль»** (между «Редактировать» и «Удалить»).

При нажатии:
1. Подтверждение: «Сбросить пароль пользователя [Имя]? Текущий пароль перестанет работать сразу.»
2. Автоматически генерируется новый временный пароль (8 буквенно-цифровых + 1 спецсимвол)
3. Открывается окно **«🔑 Пароль сброшен»** со:
   - ФИО и email пользователя
   - Новым временным паролем (выделенный код-блок + кнопка 📋 копировать)
   - ⚠️ Предупреждение «Сохраните пароль сейчас!»
4. Кнопки: **📧 Отправить на email** / **Готово**

### Логика паролей (для backend)

- Пароли хранятся в БД **только хешированными** (bcrypt или argon2)
- Plain text пароли видны HR **только в момент создания/сброса** (один раз)
- **При первом входе** сотрудник обязан сменить временный пароль на свой
- Минимальные требования к пользовательскому паролю — 8+ символов
- Старые сессии инвалидируются при сбросе пароля
- Восстановление забытого пароля — **только через HR** (сотрудник обращается, HR делает сброс)

### Учётные записи

**Операции HR:**
- создание учётки (вручную или через импорт)
- сброс пароля
- редактирование данных (ФИО, email, роли)
- блокировка / деактивация (через смену статуса сотрудника на «Неактивный» или «Архивный»)

### Системные роли

- **HR** — полный доступ к системе
- **Manager (Руководитель)** — оценка Hard Skills и СРВ своих подчинённых, 1-on-1
- **PO (Продакт-оунер)** — оценка Performance своих команд
- **Employee (Сотрудник)** — заполнение самооценки, просмотр итогов

**Множественные роли:**
- один пользователь может иметь основную роль + дополнительные
- пример: «Руководитель + Сотрудник» (тимлид, которого тоже оценивают)
- система определяет какие функции доступны по совокупности ролей

## 8.2. Справочники (master data)

HR управляет независимо от циклов:

| Справочник | Операции |
|-----------|----------|
| Направления | CRUD |
| Должности | CRUD |
| Команды / проекты | CRUD |
| Грейдовая сетка (уровни) | CRUD |

**В каждой вкладке** — кнопка «+ Добавить» с модальной формой соответствующей структуры.

## 8.3. Импорт / экспорт

**Импорт сотрудников (Excel):**
- скачиваемый шаблон с колонками (ФИО, email, должность, команда, направление, руководитель, продакт-оунер, грейд)
- валидация данных до загрузки
- режим обновления существующих vs добавления новых
- обработка «отсутствующих в файле» сотрудников (выбор действия)

**Импорт матрицы компетенций (Excel):**
- 2-листовая структура: «Матрица [Направление]» + «Описание [Направление]»
- блоки в файле: Hard Skills + СРВ (блок «Сложность решения вопросов» игнорируется)
- количество подуровней — автоматически из файла
- 4-шаговый мастер: Направление → Файл → Предпросмотр → Импорт
- предпросмотр перед сохранением (что будет импортировано)
- импорт **полностью заменяет** матрицу выбранного направления

**Экспорт:**
- отчёты по циклу — PDF и Excel
- итоговая форма по сотруднику — PDF
- выгрузка оценок по циклу — Excel
- выгрузка матрицы компетенций — Excel

## 8.4. Управление матрицей компетенций (вне цикла)

- CRUD компетенций (добавить / редактировать / удалить из направления)
- импорт компетенций из другого направления (множественный выбор)
- редактирование уровней, требований и описаний оценки
- сводная таблица внизу страницы (Итого по всем компетенциям × подуровни)

👉 Важно:
- при старте цикла создаётся снимок матрицы — дальнейшие изменения не влияют

## 8.6. Настройки цикла

При создании цикла HR через 4-шаговый мастер:

1. **Основное** — название, период
2. **Сотрудники** — выбор направлений + чекбоксы для исключения отдельных сотрудников
3. **Этапы и сроки** — дедлайны по каждому этапу
4. **Подтверждение** — чек-лист перед запуском

**Шаг «Сотрудники» — детали:**
- галочки по умолчанию включены для всех
- кнопки «Выбрать всех» / «Снять все»
- счётчик «Выбрано: N»
- HR снимает галочки с тех, кого нужно исключить

**После создания** HR может менять:
- название, описание, даты
- дедлайны этапов
- добавлять / исключать сотрудников
- переназначать оценщиков (см. 8.7)

👉 Веса оценок и минимальное количество респондентов 360 убраны (отложено вместе с конструктором и 360°).

## 8.7. Корректировки в ходе активного цикла

**Сотрудники:**
- добавить сотрудника в активный цикл
- исключить сотрудника из цикла
- перевод между командами с сохранением оценок

**Оценщики:**
- переназначить оценщика
- массовая замена оценщика

**Оценки и формы:**
- HR открывает доступ на повторную отправку формы
- ручное закрытие этапа для конкретного сотрудника

**Цикл целиком:**
- приостановка / возобновление
- отмена цикла
- архивация завершённого цикла

👉 Все корректировки логируются (см. 8.8).

## 8.8. Аудит и история (для backend)

Хотя в текущей версии прототипа раздел аудита убран из навигации, **на backend логирование сохраняется:**

- создание / запуск / завершение / отмена / удаление цикла
- продление дедлайна
- эскалации Недопуска (ПО → РУК → HR)
- переназначение оценщиков
- изменение справочников и матрицы
- смена статусов сотрудников
- вход пользователя в систему

В журнале: кто, что, когда, старое значение, новое значение.
