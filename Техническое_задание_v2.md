# 1. Цель системы

Автоматизировать процесс проведения Performance Review сотрудников, включая:

- сбор оценок от разных участников
- использование матрицы компетенций
- расчет итогов
- формирование отчетов

# 2. Общая логика процесса

Процесс запускается HR и проходит через последовательные этапы с возможностью контроля статусов.

**Основные этапы:**

- Актуализация списков участников — HR
- Актуализация списков команд - HR
- Актуализация матрицы компетенций — Лиды направлений/Руководитель
- Оценка эффективности сотрудника — Продакт-оунер
- Самооценка
- Оценка hard skills — Лиды направлений/Руковдитель
- Оценка soft skills (опрос 360) — участники команды
- Оценка CРВ (Сложность решения вопросов) – Лид направления/Руководитель
- Подведение итогов — система + HR

# 3. Роли в системе и разграничение доступа

В системе предусмотрено **4 роли**. Один пользователь может иметь несколько ролей одновременно (например Руководитель + Сотрудник). Роли назначаются глобально, но доступ к конкретным сотрудникам определяется автоматически на основе данных в карточке сотрудника.

## 3.1. HR-администратор

**Полный доступ ко всему функционалу системы.**

- Управление всеми циклами оценки
- Управление пользователями, ролями, справочниками
- Управление матрицами компетенций
- Конструктор форм
- Просмотр всех результатов и аналитики
- Доступ к аудиту и логам писем

## 3.2. Руководитель

**Видит и оценивает только своих прямых подчинённых** (определяется по полю «Руководитель» в карточке сотрудника).

**Иерархия строится автоматически:** если над Руководителем 1 есть свой Руководитель (Руководитель 2), то Руководитель 2 автоматически видит всех подчинённых Руководителя 1 + самого Руководителя 1. Дополнительной роли «Руководитель 2» не создаётся — это просто Руководитель, у которого в карточке указан вышестоящий Руководитель.

**Доступные функции:**
- Дашборд: список своих подчинённых (включая нижестоящие уровни иерархии) с прогрессом по этапам
- Оценка Hard Skills своих подчинённых
- Оценка СРВ своих подчинённых
- Карточка сотрудника + проведение 1-on-1 для своих подчинённых
- Участие в 360° оценке коллег (как обычный сотрудник)

**Недоступно:** оценка Performance (это делает Продакт-оунер), сотрудники из других направлений, админ-панель.

## 3.3. Продакт-оунер

**Видит только сотрудников своей команды** (определяется по полю «Команда» в карточке сотрудника).

**Доступные функции:**
- Дашборд: список сотрудников своей команды с прогрессом оценки эффективности
- Оценка Performance (только сотрудники своей команды)
- Принятие решения о допуске/недопуске к дальнейшим этапам оценки
- История оценок Performance

**Недоступно:** Hard Skills, СРВ, 1-on-1, самооценки, сотрудники из других команд, админ-панель.

## 3.4. Сотрудник

**Видит только свои задачи и свои результаты.**

**Доступные функции:**
- Дашборд: текущий цикл, свои этапы и дедлайны, уведомления
- Заполнение самооценки
- Участие в 360° оценке коллег (анонимно)
- Просмотр своих результатов **только после закрытия цикла HR** (итоговый балл, решение по грейду, план развития)
- История участия в прошлых циклах

**Недоступно:** результаты других сотрудников, оценка кого-либо (кроме 360°), админ-панель.

## 3.5. Комбинированные роли

Часто один пользователь имеет несколько ролей. Например:
- **Руководитель + Сотрудник** — видит свой дашборд с двумя блоками: сначала «Мои задачи» (свои формы для заполнения), потом «Мои сотрудники» (список подчинённых)
- **Продакт-оунер + Сотрудник** — аналогично, но второй блок «Сотрудники моей команды»
- **HR + Сотрудник** — переключение между админ-панелью и пользовательской частью

**Принцип:** один объединённый дашборд, на нём блоки из разных ролей в порядке приоритета (сначала задачи лично сотрудника, потом задачи по другим ролям).

## 3.6. Особенности

- Один человек может иметь несколько ролей
- Иерархия подчинения строится автоматически из данных в карточке сотрудника (поле «Руководитель»)
- Доступ к данным конкретных сотрудников определяется через связи: «Команда» (для Продакта), «Руководитель» (для Руководителя), а не через прямое назначение
- Сотрудник видит свои результаты ТОЛЬКО после того, как HR закрыл цикл

# 5. Детальная логика этапов

## Этап 1. Актуализация участников (HR)

**Функционал:**

- загрузка / редактирование списка сотрудников
- назначение:
  - продакт-оунера
  - лида
  - списка участников для 360
- возможность массового назначения

👉 Что важно учесть:

- проверка: у каждого сотрудника должен быть назначен ответственный
- нельзя перейти дальше без заполнения

## Этап 2. Актуализация матрицы (Лиды)

**Функционал:**

- просмотр/корректировка компетенций
- редактирование уровней и описаний
- выбор компетенций для оценки

👉 Важно:

- версия матрицы фиксируется на момент запуска цикла
- изменения после старта не влияют на текущий цикл

## Этап 3. Оценка эффективности (Продакт-оунер/Руководитель)

### Форма оценки

| Критерий | Описание | Оценка |
|----------|----------|--------|
| Выполнение задач | Степень выполнения поставленных задач | |
| Достижение целей / KPI | Достижение целевых показателей | |
| Качество результатов | Соответствие ожиданиям | |
| Соблюдение сроков | Выполнение в установленные сроки | |
| Общая результативность | Итоговая оценка | |
| Комментарий | Общее заключение | |
| Решение | Допуск/Недопуск | |

👉 Важно:
- В случае, если результативность низкая, может быть принято решение о недопуске сотрудника к дальнейшим этапам. В этом случае оценка не проводится.
- это отдельный блок, не связанный напрямую с компетенциями
- может иметь вес в финальном расчете

## Этап 4. Самооценка (Сотрудник)

### Общая логика

Сотрудник самостоятельно заполняет форму самооценки до начала оценки эффективности. Форма основана на фиксированном шаблоне и заполняется в рамках текущего цикла оценки.

### Структура формы самооценки

Форма состоит из трёх шагов:

#### Шаг 1. Достижения за период

Сотрудник отвечает на вопрос: «Чем гордитесь? Опишите свой вклад в развитие компании»

Поля для заполнения:
- Внедрение значимых проектов / крупных задач
- Нововведения и идеи с вау-эффектом
- Список достижений — до 6 пунктов (текстовые поля, пронумерованные)

#### Шаг 2. Выполнение плана развития

Сотрудник комментирует, что удалось выполнить из запланированных пунктов Плана развития.

Формат — таблица с до 3 строк:

| № | Навык, который развивал | Комментарий |
|---|------------------------|-------------|
| 1 | | |
| 2 | | |
| 3 | | |

#### Шаг 3. Самооценка уровня

Сотрудник оценивает собственный уровень согласно грейдовой сетке (Jun1–Sen3) и обосновывает:
- текущий уровень (выбор из сетки)
- позиция: повысить / оставить / понизить (выбор)
- обоснование (текстовое поле, обязательное)

### Важно учесть

- форма доступна сотруднику только в рамках активного цикла оценки
- сотрудник может сохранять черновик и возвращаться к заполнению
- после отправки форма недоступна для редактирования
- HR и Лид видят заполненную самооценку только после её отправки
- этап считается завершённым после отправки формы сотрудником
- нельзя перейти к следующему этапу (оценка эффективности), пока самооценка не заполнена

### Отображение для HR

- статус по каждому сотруднику: не начато / в процессе / отправлено
- дедлайн этапа с возможностью продления
- автоматическое напоминание при приближении дедлайна

## Этап 5. Оценка hard skills (Лиды)

**Форма:**

- список компетенций (hard)
- шкала (например 0-3)
- описание каждой оценки по каждой компетенции
- комментарий по каждой компетенции (опционально)

👉 Важно:

- подтягиваются только релевантные компетенции по роли

## Этап 6. Оценка soft skills (360)

### Общая логика

Форма оценки soft skills — **опросник из вопросов**. Респондент **не видит**, к какой компетенции относится вопрос — виден только текст вопроса и шкала 1–5. Каждый вопрос привязан к компетенции на уровне шаблона (скрыто от респондента). По итогу система агрегирует ответы в оценку по компетенции.

### Конструктор опросника

HR (или Лид) собирает опросник через встроенный конструктор:

**Вопрос:**
- текст вопроса
- привязка к компетенции (одна компетенция — много вопросов) — **скрыта от респондента**
- порядок отображения (sort_order)
- обязательный / необязательный
- наличие поля для комментария (да / нет)

**Шкала:**
- Единая шкала 1–5 для всех вопросов (без текстового описания вариантов)
- Респондент выбирает балл от 1 до 5

### Расчёт оценки по компетенции (маппинг)

**Шаг 1.** Респондент отвечает на вопросы по шкале 1–5

**Шаг 2.** По каждой компетенции считается среднее по всем её вопросам у одного респондента → итог по компетенции (1–5)

**Шаг 3.** Среднее по всем респондентам → финальная оценка компетенции (1–5)

**Шаг 4.** Маппинг в шкалу 0–3:

| Среднее (1–5) | Оценка компетенции (0–3) |
|---------------|--------------------------|
| 1.0 – 2.0     | 0                        |
| 2.1 – 3.0     | 1                        |
| 3.1 – 4.0     | 2                        |
| 4.1 – 5.0     | 3                        |

### Рассылка

- HR формирует список участников (кто кого оценивает)
- участникам отправляется ссылка на опросник внутри системы

### Сбор данных и обработка

Для каждого сотрудника система рассчитывает:

- среднюю оценку по каждой компетенции (на основе вопросов)
- маппинг средней оценки в шкалу 0–3
- общую оценку soft skills (сумма маппированных баллов)
- количество респондентов

👉 Учитываются только ответы от назначенных участников

### Контроль

Система отображает:

- список назначенных респондентов
- количество полученных ответов
- % заполнения по каждому сотруднику

### Завершение этапа

Этап считается завершенным:

- при достижении минимального количества ответов, или
- по дедлайну (настраивается HR)

### Важно учесть

- один респондент может оценивать нескольких сотрудников
- один сотрудник оценивается несколькими респондентами
- респондент НЕ видит привязку вопроса к компетенции
- защита от повторных ответов (1 человек = 1 ответ на одного оцениваемого)
- привязка ответов к текущему циклу оценки

## Этап 7. Подведение итогов (1-on-1)

### Общая логика

После завершения всех этапов оценки Лид направления / Руководитель проводит встречу 1-on-1 с сотрудником. В ходе встречи или после неё заполняется итоговая форма в системе. Финальное решение по грейду принимается совместно Руководителем и HR.

### Назначение встреч 1-on-1

Система должна предусмотреть возможность планирования встречи 1-on-1 с сотрудником через интеграцию с календарём (Outlook / Google Calendar / Microsoft Teams). Конкретный способ реализации (одностороннее создание события, двусторонняя синхронизация, выбор слотов и т.д.) — **на усмотрение разработчика**.

Минимально ожидается:
- возможность предложить / зафиксировать дату и время встречи из карточки сотрудника
- отображение даты встречи в итоговой форме и в дашборде HR
- автоматическое напоминание за N часов до встречи (параметр из шаблонов уведомлений, ТЗ 8.9)

|------|--------|
| Руководитель / Лид | Просмотр + заполнение комментариев и решения |
| HR | Просмотр + подтверждение решения по грейду |
| Сотрудник | Просмотр полной формы после подтверждения обеими сторонами |

### Завершение цикла

Цикл оценки считается завершённым когда:
- итоговая форма заполнена по всем сотрудникам
- решение по грейду подтверждено Руководителем и HR
- сотрудник получил доступ к своей итоговой форме

# 8. Административная часть

Раздел описывает возможности по наполнению, корректировке и управлению данными системы вне зависимости от конкретного цикла оценки, а также вмешательство в уже запущенные циклы.

В системе только HR выполняет административные функции — отдельной роли «Администратор» нет.

## 8.1. Управление пользователями системы

**Учётные записи для входа:**
- создание учётки (ФИО, email, логин)
- способ аутентификации уточняется на этапе разработки
- сброс пароля / смена email
- блокировка и деактивация пользователя (увольнение)

**Системные роли:**
- HR — полный доступ к системе, управление справочниками, шаблонами и циклами
- Пользователь — участие в циклах (оцениваемый / оценщик)

👉 Важно:
- один человек может быть одновременно и HR, и участником цикла
- при деактивации пользователя должна быть понятна судьба его оценок и назначений в активных циклах (см. 8.7)

## 8.2. Справочники (master data)

HR управляет независимо от циклов:

| Справочник | Операции |
|-----------|----------|
| Команды / проекты | создание, переименование, расформирование |
| Должности | CRUD |
| Направления / специализации (Backend, Frontend, QA, HR, маркетинг и т.д.) | CRUD |
| Роли в цикле | CRUD |

👉 Важно:
- система должна позволять добавлять новые направления и роли по мере необходимости (список не фиксирован)
- нельзя удалить элемент справочника, если он используется в активном цикле — только архивировать
- архивные элементы не отображаются при создании нового цикла, но сохраняются в истории

### Грейдовая сетка (настраиваемая)

Грейдовая сетка **полностью настраивается через интерфейс** администратора (HR).

**Структура грейдовой сетки:**
- Сетка состоит из **уровней** (например: Junior, Middle, Senior, Lead) и **подуровней** внутри каждого уровня (например: Jun1, Jun2, Jun3).
- Количество уровней и подуровней не фиксировано — HR может добавлять, переименовывать, удалять и менять порядок.

**Операции с уровнями:**
- добавление нового уровня (например: Lead, Principal, Staff)
- переименование уровня
- удаление уровня (только если он не используется в активном или завершённом цикле — иначе только архивация)
- изменение порядка отображения (drag-and-drop или стрелки вверх/вниз)

**Операции с подуровнями:**
- добавление подуровня внутри уровня (например: Lead1, Lead2)
- переименование подуровня
- удаление подуровня (с теми же ограничениями, что и для уровней)
- изменение порядка внутри уровня

**Валидации:**
- нельзя удалить уровень/подуровень, если он назначен хотя бы одному сотруднику в активном цикле
- при удалении уровня все его подуровни также удаляются / архивируются
- система предупреждает, сколько сотрудников затронет удаление (в завершённых циклах)
- минимально в сетке должен быть хотя бы один уровень с одним подуровнем

**Связь с матрицей компетенций:**
- при добавлении нового уровня/подуровня система предлагает заполнить требования к нему в матрице компетенций
- при удалении уровня требования к нему в матрице архивируются

**Дефолтная сетка при установке системы:**
- Junior (Jun1, Jun2, Jun3)
- Middle (Mid1, Mid2, Mid3)
- Senior (Sen1, Sen2, Sen3)

HR может изменить её в любой момент вне активного цикла.

## 8.3. Импорт / экспорт

**Импорт сотрудников (Excel):**
- скачиваемый шаблон с обязательными колонками (ФИО, email, должность, команда, направление)
- валидация данных до загрузки (отчёт об ошибках и дублях)
- режим обновления существующих vs добавления новых

**Импорт матрицы компетенций (Excel):**
- загрузка из файла формата «Матрица .Net BackEnd.xlsx»
- импорт компетенций, уровней, требований и описаний
- предпросмотр перед сохранением

**Экспорт:**
- отчёты по циклу — PDF и Excel
- итоговая форма по сотруднику — PDF
- выгрузка оценок по циклу — Excel
- выгрузка матрицы компетенций — Excel

## 8.4. Управление матрицей компетенций (вне цикла)

В системе **одна единая матрица компетенций** для всех направлений.

**Функционал:**
- CRUD компетенций (добавить / редактировать / архивировать)
- редактирование уровней, требований и описаний оценки
- управление таблицей «Требования по ролям» (п.4.4): текущий / ожидаемый уровень по роли
- версионирование матрицы (история изменений, возможность откатиться)

👉 Важно:
- при старте цикла создаётся снимок матрицы — дальнейшие изменения на него не влияют (см. Этап 2)
- удаление компетенции невозможно, если она использована в снимке активного или завершённого цикла — только архивация

## 8.5. Шаблоны форм (конструктор)

Доступ к конструктору форм — **только у HR**.

Единый конструктор используется для всех этапов:
- Performance (Этап 3)
- Самооценка (Этап 4) — три шага
- Hard skills (Этап 5)
- Soft skills 360 (Этап 6)
- СРВ (Этап 6)
- Итоговая форма 1-on-1 (Этап 7)

**Операции с шаблонами:**
- создание шаблона с нуля
- дублирование шаблона из прошлого цикла
- предпросмотр формы в режиме заполнения
- версионирование (история изменений шаблона)
- архивация

**Назначение шаблонов в цикле:**
- в рамках одного цикла для одного этапа могут быть назначены **разные шаблоны разным группам сотрудников** (например, отдельный 360-опросник для Backend и отдельный для маркетинга)
- группировка назначения — по направлению / команде / списку сотрудников

👉 Важно:
- при старте цикла создаётся снимок шаблона — дальнейшие изменения шаблона не влияют на активный цикл
- редактирование шаблона, использованного в активном цикле, невозможно — только клонирование и правка копии

## 8.6. Настройки цикла (после создания)

После создания цикла HR может менять:
- название, описание, даты начала / окончания
- дедлайны по каждому этапу
- **веса оценок** (Performance / Hard / Soft / СРВ) для итогового балла
- минимальное количество респондентов 360 для завершения этапа
- назначение шаблонов форм для этапов (с учётом группировки из 8.5)

👉 Важно:
- изменение весов в уже завершённом цикле пересчитывает итоговый балл (требует подтверждения)
- дедлайн можно продлевать с фиксацией в аудите (см. 8.8)
- в системе одновременно может быть только **один активный цикл** на всех сотрудников

## 8.7. Корректировки в ходе активного цикла

Это ключевой блок — реальность всегда вносит правки в уже запущенный процесс.

**Сотрудники:**
- добавить сотрудника в активный цикл (пришёл после старта)
- исключить сотрудника из цикла (уволился, ушёл в долгий отпуск)
- перевод сотрудника между командами с сохранением оценок

**Оценщики:**
- переназначить оценщика (болезнь / увольнение / отпуск)
- массовая замена оценщика (все оценки Иванова перебросить на Петрова)
- добавить / убрать респондента 360

**Оценки и формы:**
- **HR открывает доступ на повторную отправку формы** (по запросу пользователя, с указанием причины) — отправленная форма становится снова редактируемой
- сбросить ответ респондента 360 (при ошибочной отправке)
- ручное закрытие этапа для конкретного сотрудника (например, «допуск» при спорной ситуации)

**Цикл целиком:**
- приостановка / возобновление цикла
- отмена цикла (перевод в архив без завершения)
- **архивация завершённого цикла** (read-only доступ)
- **физическое удаление цикла** (по решению HR, с подтверждением)
- клонирование цикла (создать следующий квартал на базе текущего — см. 8.12)

👉 Важно:
- все корректировки логируются в аудит с указанием причины
- разблокировка формы доступна только HR
- исключённые сотрудники остаются в истории, их оценки не стираются
- физическое удаление цикла возможно только для draft или отменённых циклов, действие необратимо

## 8.8. Аудит и история

Система ведёт журнал действий:
- создание / запуск / завершение / отмена / удаление цикла
- продление дедлайна
- разблокировка и сброс форм
- переназначение оценщиков
- изменение весов и шаблонов
- изменение справочников и матрицы
- вход пользователя в систему

**В журнале фиксируется:** кто (реальный пользователь, даже если действие выполнено в режиме делегирования — см. 8.10), что, когда, старое значение, новое значение, причина (для критичных действий).

👉 Каждое действие в системе должно быть закреплено за конкретным пользователем. Даже когда один HR передаёт полномочия другому, в аудите виден реальный исполнитель.

👉 Доступ к аудиту — только HR.

## 8.9. Уведомления

**Канал — только email.**

**Шаблоны:**
- редактируемый текст шаблонов (с подстановкой переменных: имя сотрудника, название цикла, дедлайн, ссылка)
- версионирование шаблонов

**Правила рассылки:**
- событие-триггер (назначение оценки, приближение дедлайна, просрочка, завершение этапа)
- периодичность напоминаний (за N дней до дедлайна, далее каждый день)

**Пользовательские настройки:**
- пользователь может включать / отключать отдельные категории уведомлений

## 8.10. Права доступа и делегирование

**Видимость данных:**
- Продакт-оунер видит только свои оценки и сотрудников из своих команд
- Лид видит hard skills и СРВ только своего направления
- сотрудник видит только свою итоговую форму после подтверждения
- HR видит всё

**Делегирование прав HR:**
- временная передача прав HR другому пользователю (отпуск, командировка) с указанием срока
- в аудите фиксируется **реальный исполнитель** действия, а не роль (см. 8.8)
- по окончании срока делегирование автоматически отзывается

## 8.11. Архивация и хранение данных

- завершённые циклы переводятся в архив с доступом только на чтение
- **срок хранения оценок задаётся в настройках системы** (например, 3 года после завершения цикла); по истечении срока данные могут быть удалены или обезличены
- HR может удалить данные вручную раньше срока (с подтверждением и записью в аудит)

## 8.12. Клонирование цикла

При создании нового цикла HR может скопировать данные из предыдущего с возможностью выбора что именно переносить:

- список участников
- назначения оценщиков (продакт-оунер, лид, респонденты 360)
- веса оценок
- дедлайны (со сдвигом на новый период)
- шаблоны форм для этапов
- минимальные пороги респондентов

Перед созданием отображается предпросмотр — какие данные будут перенесены.

## 8.13. Массовые операции

HR-операции, охватывающие сразу нескольких сотрудников:
- массовое назначение оценщиков по правилам (по команде / направлению)
- массовое продление дедлайна
- массовая рассылка напоминаний опоздавшим
- массовое закрытие этапа

## 8.14. Интеграция с календарём Microsoft Teams

Система интегрируется с календарём Teams для планирования встреч:

**1-on-1 встречи (Этап 7):**
- создание события в календаре Teams напрямую из карточки сотрудника
- автоматическое приглашение сотрудника и руководителя
- привязка ссылки на встречу к итоговой форме

**Встречи по оценке hard skills (Этап 5):**
- лид / руководитель может запланировать встречу с сотрудником для очной оценки hard skills через календарь Teams
- автоматическое приглашение сотрудника
- привязка события к форме оценки hard skills

👉 Важно:
- интеграция выполняется через Microsoft Graph API (уточнить на этапе разработки)
- перенос / отмена встречи синхронизируется с системой

## 8.15. Управление назначениями оценщиков

HR управляет тем, **кто оценивает кого и на каком этапе** в рамках конкретного цикла. Назначения хранятся в `evaluation_assignments` (связь «оценщик × оцениваемый × тип оценки / этап»).

**Где отображается:**
- отдельный экран **«Назначения оценщиков»** — матрица назначений по всему циклу, доступная HR;
- карточка сотрудника на экране «Участники цикла» — раскрывающийся список его оценщиков по этапам.

**Что можно делать:**

1. **Смотреть матрицу назначений** — таблица вида «сотрудник × этап», в каждой ячейке — ФИО назначенного оценщика или метка «не назначен».
2. **Фильтровать**:
   - по направлению (.NET Backend, Frontend, Mobile, Data, QA, DevOps);
   - по команде / проекту;
   - по этапу (Hard skills, Soft 360°, Performance, CRV, 1-on-1);
   - по статусу назначения («не назначен», «назначен», «в работе», «завершено»).
3. **Назначать оценщика одному сотруднику** — inline-редактирование ячейки с выбором из справочника участников цикла, имеющих соответствующую роль.
4. **Массово назначать по направлению / команде** — модал «Назначить по направлению»:
   - выбрать направление / команду + этап + оценщика;
   - применить ко всем сотрудникам направления, у кого нет назначения или принудительно перезаписать;
   - предпросмотр (сколько сотрудников затронет) и подтверждение.
5. **Массово заменять оценщика** — см. 8.13: перенос всех назначений одного оценщика на другого (при увольнении / делегировании).
6. **Экспортировать матрицу** в Excel / CSV для проверки перед запуском этапа.

**Валидации:**
- нельзя назначить оценщика, не имеющего соответствующей роли в цикле (`participant_roles`);
- для Soft 360° — не менее 3 и не более N (N задаётся в настройках цикла) оценщиков на сотрудника;
- нельзя назначить сотрудника оценивать самого себя;
- предупреждение, если оценщик и оцениваемый в одной команде и разница по уровням превышает допустимую;
- для Performance — только продакт-оунер или руководитель сотрудника;
- для Hard skills / CRV — только лид направления или руководитель.

**Работа в запущенном цикле:**
- изменение назначения в активном этапе допускается, но требует причины;
- если у оценщика уже есть заполненный черновик — при замене черновик сохраняется, новому оценщику выдаётся чистая форма;
- все изменения попадают в Аудит (см. 8.8);
- затронутые оценщики получают уведомление (email + Teams, см. 8.9).

**Автоматическое назначение при старте цикла:**
- система предлагает дефолтные назначения на основе оргструктуры:
  - Hard skills и CRV → лид направления сотрудника;
  - Performance → продакт-оунер команды;
  - Руководитель → линейный менеджер сотрудника;
  - Soft 360° → HR пересобирает вручную или импортирует список респондентов;
- HR может принять предложение или скорректировать до запуска этапа.
